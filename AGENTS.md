## Review Protocol

Если ты выступаешь ревьювером (запущен через codex-review workflow):
- Давай конкретный actionable фидбек
- Можешь смотреть код/diff самостоятельно
- Не запускай скрипты из skills/codex-review/ — ты ревьюер
- Не заглядывай в .codex-review/archive/ — там артефакты прошлых сессий
- После ревью запиши вердикт по пути, указанному в промпте ревью (одно слово: APPROVED или CHANGES_REQUESTED)

## Skill Implementation Preferences

- Для новых скиллов по возможности предпочитай простые `.sh`-скрипты вместо сложных библиотек и тяжёлого стека
- Если shell-скриптам не хватает удобного вывода или нужно беречь контекст, кешируй результаты в директории скилла по примеру `yandex-metrika`
- Если без Python не обойтись, сначала смотри в сторону стандартной библиотеки и простого запуска через `uv`, чтобы минимизировать зависимости
- Если какой-то код вызывается часто, много и упирается в производительность, допускается вынести горячий путь в Go-бинарник и поставлять его вместе с релизом репозитория
